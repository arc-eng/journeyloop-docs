    Engineering Process

    How the JourneyLoop agent team manages issues, specs, and development workflow.

    Repos

    Repo Purpose Short Ref
    arc-eng/journeyloop Django app (code) JL
    arc-eng/journeyloop-startup-assistant Planning, roadmap, specs JL-SA

    Planning Docs: One Folder Per Feature

    Every feature gets a folder under its project namespace:

    planning/<project>/<feature>/
  brief.md          ‚Üê Product spec: user stories, acceptance criteria
  ux.md             ‚Üê UX concept: layout, interaction, wireframes
  tech-spec.md      ‚Üê Technical architecture: system design, decisions

    Not every feature needs every file

    Backend-only features skip ux.md. Simple features might only need brief.md.

    Example structure:

    planning/companion/
  memory-system/
    brief.md
    tech-spec.md
  interface/
    brief.md
    ux.md
    tech-spec.md
  skill-system/
    brief.md
    ux.md
    tech-spec.md

    Issues: One Per Feature

    Each feature has one tracking issue in arc-eng/journeyloop-startup-assistant. The substance lives in the planning folder ‚Äî the issue is lightweight.

    Issue Format

    ## [emoji] Feature Name

**Planning:** `planning/<project>/<feature>/`

| Vertical     | Status         |
|-------------|----------------|
| Product Spec | ‚úÖ Done / üî≤ Not started / ‚ûñ N/A |
| UX Concept   | ‚úÖ Done / üî≤ Not started / ‚ûñ N/A |

### Summary
One paragraph describing the feature and why it matters.

    The status table is updated as verticals are completed.

    Lifecycle: Feature State Machine

    Each feature issue carries exactly one lifecycle label at a time. The label answers: "What does this feature need next?"

    stateDiagram-v2
    [*] --> needs_spec : Feature created
    needs_spec --> needs_ux : PM writes brief
    needs_ux --> needs_tech_spec : UX concept done
    needs_tech_spec --> needs_review : CTO writes tech spec
    needs_review --> ready_for_dev : Marco approves
    ready_for_dev --> [*] : SWE ships it

    needs_spec : needs-spec
    needs_ux : needs-ux
    needs_tech_spec : needs-tech-spec
    needs_review : needs-review
    ready_for_dev : ready-for-dev

    Lifecycle Labels

    Label Set By Means
    needs-spec ‚Äî Feature needs a product spec (brief.md)
    needs-ux PM Brief done, needs UX concept
    needs-tech-spec PM/UX Ready for CTO to write tech-spec.md
    needs-review CTO All specs done, needs Marco's review
    ready-for-dev Marco Fully specced and approved, SWE can implement
    needs-po-input CTO CTO needs product clarification
    needs-cto-input PM PM needs technical assessment
    blocked-technical CTO Technical blocker identified

    Escalation Labels (to Marco)

    Escalation labels trigger a ping to Marco

    Label Action Means
    needs-founder-decision Ping Marco Above agent authority ‚Äî options + recommendation
    needs-founder-approval Ping Marco Both agents signed off, need go/no-go
    founder-fyi Silent Awareness only

    Agent Roles

    Role Owns Transitions
    PM brief.md, acceptance criteria, prioritization needs-spec ‚Üí needs-ux or needs-tech-spec
    CTO tech-spec.md, architecture decisions, PR review needs-tech-spec ‚Üí needs-review
    UX Agent ux.md, interaction design, wireframes needs-ux ‚Üí needs-tech-spec or needs-review
    Marco Founder decisions, approvals needs-review ‚Üí ready-for-dev
    SWE Implementation, PRs ready-for-dev ‚Üí done

    Label Taxonomy

    feature ¬∑ bug ¬∑ tech-debt ¬∑ enhancement ¬∑ adr

    backend ¬∑ frontend ¬∑ api ¬∑ cli ¬∑ calendar ¬∑ ai

    size/S ¬∑ size/M ¬∑ size/L ¬∑ size/XL

    p0-critical ¬∑ p1-high ¬∑ p2-medium ¬∑ p3-low

    vision ¬∑ bug ¬∑ enhancement ¬∑ epic

    Documentation Rules

    Every label change gets a comment

    Explain WHY the label changed and WHO needs to act next.

    Comment signing

    All agents sign their comments:

    • ‚Äî PO Agent
    • ‚Äî CTO Agent
    • ‚Äî UX Agent
    • ‚Äî SWE Agent

    Specs are files, not comments

    Never paste spec content into issue comments

    Tech specs, UX concepts, and briefs are files in the planning folder. Issue comments are transition signals only ‚Äî one line linking the file. Pasting specs into comments creates drift if the file changes.

    Tech spec written: `planning/companion/interface/tech-spec.md`
Removing `needs-tech-spec`, adding `needs-review`.
‚Äî CTO Agent

    ## Tech Spec
### Architecture
We will use Django REST framework to...
[2000 words of spec content pasted inline]
‚Äî CTO Agent

    Acceptance criteria use checkboxes

    ## Acceptance Criteria
- [ ] CompanionMemory model created with required fields
- [ ] Migration generated and tested
- [ ] Unit tests for model validation

    When checking off criteria, comment what you did:

    ‚úÖ Checked off "Add companion memory model"
‚Äî Added CompanionMemory in companion/models.py with migration 0001.
‚Äî SWE Agent

    Epics

    Epics group related features. They're checklist issues ‚Äî no specs or acceptance criteria of their own.

    ## [emoji] Epic Name
Why this matters ‚Äî 1-2 sentences.

### Features
- [ ] #XX ‚Äî Feature Name (`current-label`)
- [ ] #YY ‚Äî Feature Name (`current-label`)
- [x] #ZZ ‚Äî Feature Name (`done`)

    Max 2 active epics at any time.

    Implementation Issues

    When a feature reaches ready-for-dev, implementation issues are created in arc-eng/journeyloop (the code repo):

    **As a** [role],
**I want** [goal]
**so that** [value].

## Acceptance Criteria
- [ ] Testable condition (max 3 per issue ‚Äî split if more)
